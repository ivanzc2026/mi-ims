Category 2 — Status & Timeline
9 fields Where is this implementation in the lifecycle? These fields drive all automation triggers.
Field name Field type Status Priority Source of data Notes / builder guidance
Implementation Status
The lifecycle stage this implementation is currently in
 Single select New Must Ivan manually sets; some transitions auto-trigger via Zapier Options (in order): Committed → Training In Progress → Training Complete → Launch Ready → Active → Showcase Prep → Closed → Renewed. This is distinct from the existing "Site Status" field which tracks partnership status (Confirmed + Launched, Stalled Before Launch, etc.). Both fields should coexist — Site Status tracks the partner relationship; Implementation Status tracks the operational lifecycle. Do not merge them.
Site Status
Partnership relationship status (existing field)
 Single select ✓ Exists — keep Must Ivan / Farah maintain Keep as-is. Current values (Confirmed + Launched, Stalled Before Launch, Confirmed - not started) are useful for partnership-level reporting. Add "Stalled After Launch" if not already present. Do not repurpose this field for the operational lifecycle — that's what Implementation Status is for.
Current Tier
Green / Yellow / Red — updates dynamically from pulse data
 Single select New Must Auto-calculated from Weekly Pulse Log via Airtable formula or Zapier Options: Green / Yellow / Red. Logic: Green = on-time pulses + score ≥ 3 + deliverables on track. Yellow = 2 consecutive score ≤ 2, OR entering M3 (auto), OR 1 missed submission. Red = score ≤ 2 in M2/M3 window, OR module deliverable missing at module end, OR 2+ consecutive missed submissions. This field drives the Monday digest sort order (Red first).
Open Flag
Boolean — set when a risk signal fires; cleared by Ivan when resolved
 Boolean New Must Auto-set by Airtable automation or Zapier when risk threshold crossed; manually cleared by Ivan This field powers the "needs attention" section of the Implementation Control Center dashboard. Any record with Open Flag = true surfaces at the top of Ivan's Monday view. Ivan clears the flag after resolving the issue. Can be a checkbox field in Airtable.
Cycle
Which program cycle (Fall 2026, Summer 2026, etc.)
 Single select ✓ Exists Must Ivan assigns at record creation Keep as-is. Already used well for grouping and reporting. Ensure new cycles are added before the cycle begins.
Start Date
First session date
 Date ✓ Exists Must Pacing mapper output; confirmed by teacher Exists — ensure it's populated from the pacing mapper, not estimated manually. This field drives the T-minus countdown logic.
Client Kickoff Date
Date of the in-person or virtual client kickoff session with students
 Date New Must Pacing mapper → client confirmation → Ivan logs Does not currently exist as a separate field. The SY25-26 data shows in-person client kickoff is one of the strongest positive predictors of implementation quality. This date must be confirmed before launch gate closes. Distinct from Start Date.
Midpoint Date
Date of the midpoint client feedback session (M5)
 Date New Must Pacing mapper → client confirmation → Ivan logs Required for Stage 6 automation triggers (T-14 employer check-in fires based on this date). Must be confirmed at Stage 3 before launch gate can close.
Showcase Date
Date of the final student presentation to the client
 Date New Must Pacing mapper → client confirmation → Ivan logs The most important date field in the system. Drives: T-21 showcase prep trigger, T-10 check-in, T-1 reminder, post-showcase survey automation, and client debrief scheduling. Must be confirmed at Stage 3.
Category 3 — Training Tracking
5 fields All new. These fields enable the training automation chain and the readiness checkpoint flagging.
Field name Field type Status Priority Source of data Notes / builder guidance
Part 1 Completion Date
Timestamp when teacher completes Reach 360 Part 1
 Date / Timestamp New Must Reach 360 course.completed webhook → Zapier → Airtable (confirmed supported) This field is the trigger for the Part 2 window selection email. If it's populated, Part 2 selection email has already fired. If it's empty at T-21, Ivan gets a risk alert. The webhook integration is confirmed — this field can be auto-populated with zero manual work.
Part 2 Window Selected
Date of the Part 2 live session the teacher has booked
 Date New Must Calendly booking → Zapier → Airtable Auto-populated when teacher books via Calendly. If empty at T-18 after Part 1 completion, automated nudge fires. If still empty at T-16, Ivan text. The date here tells Ivan which window the teacher chose — useful for scheduling the onboarding call after Part 2.
Part 2 Attended
Confirms teacher completed the live PD session
 Boolean New Must Ivan manually checks after session; or sign-in form → Zapier Simple checkbox. Cannot be automated as reliably as Part 1 (no native Calendly "attended" webhook). Ivan marks true after the session. If false and Part 2 Window date has passed, Zapier fires re-offer email automatically.
Readiness Checkpoint Score
Lowest score from the 5-question readiness self-check (1–5), weighted toward M2 and M3 facilitation confidence
 Number (1–5) New Must Google Form → Zapier → Airtable (auto-populated) Store the lowest score from the two most critical questions (M2 interview coaching confidence, M3 synthesis facilitation confidence). If score ≤ 2, auto-flag fires to Ivan. This is the closest proxy for "teacher is likely to struggle" available before launch. Consider also storing the raw text from the optional open-field question for Ivan's Monday review.
Onboarding Call Completed
Ivan + teacher 30-min call done; status can advance to Training Complete
 Boolean New Must Ivan manually checks after call Simple checkbox. Ivan checks this after the onboarding call, then manually flips Implementation Status to Training Complete. These two actions are intentionally linked — the status change should not happen before the call. Consider an Airtable automation that validates: Onboarding Call Completed must be true before Implementation Status can be set to Training Complete.
Category 4 — Active Implementation
4 fields All new. These are the heartbeat of Stage 5 — the fields that make the pulse form, tier logic, and JIT triggers work.
Field name Field type Status Priority Source of data Notes / builder guidance
Current Session
Which session (1–30) the teacher ran most recently
 Number (1–30) New Must Weekly pulse form → Zapier → Airtable (auto-populated); H2: Chipp Bot interaction This field drives the module transition detection logic. When Current Session crosses 6, 11, 16, or 20, Zapier fires the corresponding JIT kit. The field also enables the "Active by Module" Airtable view — grouping schools by which module they're in. Rollup in the Program Implementation table from the linked Weekly Pulse Log (latest record's session number).
Last Pulse Submission Date
Timestamp of the most recent weekly pulse form submission
 Date / Timestamp New Must Google Form → Zapier → Airtable (auto-populated from pulse form submission timestamp) The missed-submission detection logic depends entirely on this field. If today is Saturday and Last Pulse Submission Date is more than 7 days ago, the Saturday reminder fires automatically. Rollup from the linked Weekly Pulse Log table (max date of linked records).
Latest Pulse Score
The 1–5 confidence score from the most recent pulse submission
 Number (1–5) New Must Rollup from Weekly Pulse Log (most recent record's score field) Used in tier logic formula. If Latest Pulse Score ≤ 2 AND Current Session is in range 6–15 (M2/M3 window), trigger Red tier automatically. This field alone is not enough — tier logic also requires looking at the previous week's score (to detect 2 consecutive low scores). Store last two scores in the Weekly Pulse Log and use a rollup formula here.
Module Deliverable Status
Whether the current module's required deliverable has been submitted on time
 Single select New Must Pulse form Y/N question at module end → Zapier → Airtable; H2: Chipp Bot Drive monitoring Options: On Track / Missing / Flagged. "Missing" = deliverable not submitted by the end-of-module session. "Flagged" = submitted but quality concern noted by Ivan. This field triggers a Red flag automatically when set to Missing. It resets to On Track at the start of each new module. Consider whether to track per-module or just current status — for v1, current status is sufficient.
Category 5 — Employer / Partner
4 fields Mix of existing and new. The MI Client Partner table already exists — these fields extend it rather than duplicate it.
Field name Field type Status Priority Source of data Notes / builder guidance
MI Client Partner
Linked record to the MI Client Partner table
 Linked record ✓ Exists Must Linked to MI Client Partner table (separate table, already exists) The MI Client Partner table already exists in the Sandbox Base with 32 clients tracked. Keep the linked record relationship. The client's contact information, industry, partner type, and client success factors already live there. Don't duplicate fields — extend the existing table instead.
Employer Brief Status
Where is the client brief in the creation and approval process
 Single select New Must Farah updates as brief moves through creation → review → approval Options: Not Started / Draft / Sent to Client / Approved. This field is one of the 7 launch gate items — launch cannot proceed until Employer Brief Status = Approved. Currently Farah creates briefs from a template; this field makes that process trackable without changing the process itself.
Client Kickoff Confirmed
Boolean — client has confirmed they will attend the kickoff session
 Boolean New Must Ivan logs when client confirms; email confirmation is the evidence Separate from Client Kickoff Date (which is when the kickoff is). This boolean confirms the client has agreed to attend. Required for the launch gate. SY25-26 data: in-person client kickoff is one of the strongest positive predictors of implementation quality — this field protects that condition.
Client Engagement Quality
Post-implementation assessment of how engaged and rigorous the client actually was
 Single select New Should Ivan fills in at closeout based on observation and teacher debrief The existing "Client Quality" field in the Sandbox Base uses values like "Highly Engaged and Rigorous" and "Showed Up / Low Rigor." Use those same values — this is just making that field available at the implementation level, not just the client level. The SY25-26 analysis found that "Showed Up / Low Rigor" clients appeared frequently in Average and Weak implementations. Tracking this at the implementation level (not just the client level) captures context-specific variance.
Category 6 — Closeout & Renewal
7 fields Mix of existing (already tracked, keep) and new (not tracked, add now).
Field name Field type Status Priority Source of data Notes / builder guidance
Student Survey Completion Rate
Percentage of enrolled students who completed the end-of-program survey
 Percent / Formula New Must Auto-calculated: survey response count (from form) ÷ Total # Students Served (existing field) The existing "Total # Students Served" field already exists. Survey response count can be pulled from Google Forms via Zapier and logged in a linked field. Formula: responses ÷ students enrolled. Flag if below 70% at Day 7 post-showcase → Ivan text to teacher. This is one of the most important quality indicators in the system.
Facilitator Strength
Post-implementation assessment of teacher's facilitation quality
 Single select Fix: typo Must Ivan fills in at closeout Field exists and is actively used in the SY25-26 analysis — keep it. Fix the "Stong" typo to "Strong" in the select options. Values: Weak / Average / Strong / Cambiar Facilitated. This field has real predictive value when consistently populated — do not let it become optional.
Implementation Outcome
Overall quality rating of the implementation (Strong / Average / Weak)
 Single select ✓ Exists Must Ivan fills in at closeout based on full picture Keep as-is. This is the primary outcome field used in the SY25-26 analysis. It should be populated for every implementation before status = Closed. Consider adding a validation rule: status cannot be set to Closed unless Implementation Outcome is populated.
Implementation Barriers
Multi-select of barriers that affected this implementation
 Multi-select ✓ Exists Must Ivan fills in at closeout Keep as-is. Current values (New teacher to PBL, Scheduling difficulties, Low attendance, Low student buy-in, etc.) are well-calibrated from the SY25-26 analysis. These values directly predict outcome quality — the more of them present, the more likely a Weak outcome. Populate consistently for every implementation.
MI Implementation Success Factors
Multi-select of conditions that supported this implementation
 Multi-select ✓ Exists Must Ivan fills in at closeout Keep as-is. Current values (Facilitator buy-in, Student opt-in, In person client kick off, School year activities mapped in advance, etc.) are directly validated by the SY25-26 survey analysis. These are the conditions to protect. Populate every implementation.
Key Implementation Notes
Free-text post-mortem narrative for this implementation
 Long text ✓ Exists Must Ivan writes at closeout; may include client debrief notes Keep as-is. The existing notes in the base are useful — several contain specific insights about what drove quality or failure. H2: Admin Chipp Bot drafts this field from pulse history and Drive signals; Ivan edits and confirms.
Renewal Disposition
Whether the school / network wants to run MI again
 Single select New Must Ivan logs within 24h of client debrief call Options: Yes / No / Maybe / Not Contacted. This field currently doesn't exist — there's no way to track renewal rates across cycles. Adding it enables: 30/60-day follow-up automations for Yes/Maybe; network-level renewal summary views; pattern detection (same network with 2+ No dispositions = systemic flag for Farah). Must be populated before status = Closed.
Secondary Table — Weekly Pulse Log
New table · 6 fields One record per weekly pulse submission per school. Linked to Program Implementation. Enables trend analysis and tier history.
Field name Field type Status Priority Source of data Notes / builder guidance
Implementation Key
Links this pulse record to the parent implementation
 Linked record New table Must Zapier auto-populates from Google Form "which school" dropdown This linking field connects pulse records to the parent Program Implementation record. The "which school" field in the pulse form must use a value that Zapier can match to an existing Airtable record (Implementation Key or a standardized school name dropdown). Pre-populate the form dropdown from Airtable to prevent matching errors.
Submission Date
When this pulse was submitted
 Date / Timestamp New table Must Google Form submission timestamp → Zapier → Airtable Auto-populated. Used to detect missed submissions (if no record exists for a given week, the school missed). The Saturday reminder trigger checks: does this implementation have a pulse record with Submission Date in the current week? If not, fire reminder.
Session Number
Which session (1–30) the teacher ran this week
 Number New table Must Pulse form dropdown → Zapier → Airtable Drives module transition detection. When this value crosses 6, 11, 16, or 20 relative to the previous week's record, Zapier fires the corresponding JIT kit. Also populates the "Current Session" rollup field in the parent Program Implementation record.
Pulse Score
Teacher's 1–5 confidence / quality rating for this week
 Number (1–5) New table Must Pulse form 1–5 scale → Zapier → Airtable The tier logic formula looks at the two most recent pulse records' scores. If both are ≤ 2, Yellow tier. If score ≤ 2 AND session is in M2 or M3 range, Red tier. Rollup to Latest Pulse Score field in parent record.
Open Text Response
Teacher's free-text answer to "what's your biggest question or concern?"
 Long text New table Should Pulse form optional text field → Zapier → Airtable The highest-signal data in the entire pulse form — a teacher typing "students can't find anyone to interview" tells you more than a score of 2. Ivan reviews these manually in H1; pastes the week's responses into Claude for a 3-bullet summary each Monday. H2: Admin Chipp Bot surfaces these in the daily digest automatically.
Module Deliverable Submitted
Y/N — did students complete the module deliverable? (Only appears on pulse form at module end sessions)
 Boolean New table Must Pulse form conditional Y/N question → Zapier → Airtable This field only appears on the pulse form when Session Number is at a module-end point (S5, S10, S15, S19, S24). If false at those sessions, Zapier triggers a Red flag regardless of the confidence score. Rollup populates the Module Deliverable Status field in the parent record.

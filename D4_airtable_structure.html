Quest MI — Airtable Structure Recommendation

How the Quest Base should be organized: tables, views, relationships, and the Implementation Control Center interface.
Designed for the Horizon 1 fall build. Assumes Enterprise plan (confirmed).

Architecture decision: one base, multiple tables, stage-organized views

The right answer for v1 is not separate bases per stage — that creates linking complexity and makes cross-stage reporting impossible. The right answer is one base (the existing Quest Base) with extended tables and purpose-built views that organize data by stage. The "massive Airtable" concern is a view design problem, not a structure problem. Views are free — use them aggressively to slice the same data for different purposes. Tables should only be added when data has a fundamentally different structure (e.g., one-record-per-week for pulse data vs. one-record-per-implementation).

Why a separate table (not fields on the main record)?

Storing all pulses in a separate linked table enables trend analysis — you can see whether a school's score dropped over 3 weeks, whether they recovered after an intervention, and exactly which sessions were submitted vs. missed. A single "Latest Pulse Score" field on the main record would overwrite history. History is what makes the system intelligent over time.

Relationship to Program Implementation

One client → many implementations (a client may participate across multiple cycles or schools). The linked record relationship already exists. Don't duplicate client info on the implementation record — link to it.

Existing tables — keep as-is

Network Partners + Implementing Sites

Both already exist and are structured correctly. No changes needed for v1.

Network Partners table

One record per network organization (Green Dot, LAUSD, College Track, etc.). Links to multiple Program Implementation records. The network decision-maker contact field on the implementation record references this level — consider adding it to the Network Partners table as well so it doesn't need to be re-entered per implementation.

Implementing Sites table

One record per school building. Distinct from Network Partner — Green Dot is the network; Animo City of Champions is the site. This separation is already correct in the base and should be preserved.

Recommended views — Program Implementation table

Views are free — use them aggressively. Each view below is designed for a specific use case. The same record appears in multiple views; no data is duplicated.

Ivan's Monday Dashboard
Grid view
Ivan's primary weekly triage view. Opens every Monday morning. Shows every active implementation grouped by Current Tier (Red first, then Yellow, then Green). Only shows what needs attention — not the full data set.
Filter: Implementation Status = Active OR Showcase Prep | Group by: Current Tier | Sort: Open Flag (true first), then Last Pulse Submission Date (oldest first) | Hidden fields: everything except School, Current Session, Current Tier, Last Pulse Date, Next Milestone, Open Flag, Latest Pulse Score
Training Queue
Grid view
Shows every implementation currently in training, sorted by days to launch. Ivan uses this to see who is at risk of missing the T-21 or T-14 deadlines and to prioritize direct outreach.
Filter: Implementation Status = Training In Progress | Sort: Start Date (ascending) | Key columns: School, Part 1 Completion Date, Part 2 Window Selected, Part 2 Attended, Readiness Checkpoint Score, Onboarding Call Completed, Days to Launch (formula field)
Launch Readiness Queue
Grid view
Shows every implementation at Launch Ready status, awaiting Ivan's go/no-go review. Displays the 7 launch gate checkboxes side by side so Ivan can see at a glance what's holding each school back.
Filter: Implementation Status = Launch Ready | Key columns: School, all 7 launch gate checkboxes, Showcase Date, Client Kickoff Date | Sort: Showcase Date (ascending)
Active by Module
Grid view
Shows all active implementations grouped by which module they're currently in (derived from Current Session). Ivan uses this to see how many schools are in M3 at any given time — the highest-risk module — and to spot patterns across the cohort.
Filter: Implementation Status = Active | Group by: Current Module (formula field: IF Session ≤ 5, "M1", IF Session ≤ 10, "M2", etc.) | Sort within group: Current Tier (Red first) | Key columns: School, Current Session, Latest Pulse Score, Module Deliverable Status, Current Tier
Showcase Pipeline
Calendar view
Calendar view of all upcoming showcase dates. Ivan can see at a glance when showcases are clustered and plan T-10 calls accordingly. Also useful for identifying weeks where multiple Red/Yellow sites are showcasing simultaneously.
Date field: Showcase Date | Color by: Current Tier (Red = red, Yellow = yellow, Green = green) | Filter: Showcase Date is not empty AND Implementation Status ≠ Closed AND Implementation Status ≠ Renewed
Showcase Prep Checklist
Grid view
Shows all implementations in Showcase Prep status with their prep checklist items visible. Ivan uses this during the T-21 to T-1 window to track which sites have completed which prep steps.
Filter: Implementation Status = Showcase Prep | Sort: Showcase Date (ascending) | Key columns: School, Showcase Date, Client Availability Reconfirmed, T-10 Check-in Completed, Rubric Shared, Current Tier
Closeout Queue
Grid view
Shows all implementations where the showcase has passed but status is not yet Closed. Ivan uses this to track outstanding surveys, debrief calls, and post-mortem fields. If a record stays in this view for more than 14 days, something is falling through the cracks.
Filter: Showcase Date is in the past 30 days AND Implementation Status ≠ Closed AND Implementation Status ≠ Renewed | Key columns: School, Showcase Date, Student Survey Completion Rate, Client Debrief Completed, Post-mortem fields empty (formula: boolean), Renewal Disposition
Renewal Tracker
Grid view
All closed implementations with their renewal disposition, grouped by network. Farah uses this for recruitment conversations. Ivan uses it to spot networks with multiple "No" dispositions.
Filter: Implementation Status = Closed OR Renewed | Group by: Network Partner | Sort: Renewal Disposition (No first for attention, then Maybe, then Not Contacted, then Yes) | Key columns: School, Network Partner, Renewal Disposition, Implementation Outcome, Facilitator Strength
All Data (default)
Grid view
The existing "All Data" view — keep it as the base view. Group by Cycle as currently configured. This is for historical reference and data integrity checks, not day-to-day operational use.
No filter | Group by: Cycle | All fields visible | This is the view the team uses for reporting, not for implementation management
Implementation Control Center — Airtable Interface

Ivan's Control Center (Airtable Interface page)

A single Airtable Interface page that Ivan opens every morning. Not a replacement for the views above — a summary layer on top of them. Built using Airtable's Interface Builder (available on Enterprise plan).

Row 1 — Stat blocks (summary counts)
Five metric cards across the top of the page, each pulling a count from the implementation base.
Active implementations · Red sites this week · Yellow sites this week · Missing pulses this week (no submission in last 7 days) · Showcases in next 30 days
Row 2 — Alert section (needs attention now)
A filtered grid showing only implementations with Open Flag = true, sorted by severity (Red tier first). This is the first thing Ivan looks at. If it's empty, nothing is on fire. Each row links directly to the full implementation record.
Fields shown: School, Current Tier, Current Session, Last Pulse Date, Open Flag reason (from a Notes field Ivan sets when flagging)
Row 3 — Module risk section (entering high-risk windows)
A filtered grid showing implementations where Current Session = 6–10 (entering M2) or 11–15 (in M3). These are auto-elevated to Yellow even if their score is fine — Ivan sees them here as a proactive heads-up, not as a crisis.
Fields shown: School, Current Session, Current Module (formula), Current Tier, Latest Pulse Score, Module Deliverable Status
Row 4 — Upcoming showcases (next 21 days)
A filtered grid of all implementations with Showcase Date in the next 21 days. Shows prep checklist completion status at a glance. Ivan uses this to confirm T-10 calls are scheduled.
Fields shown: School, Showcase Date, T-10 Check-in Completed, Client Availability Reconfirmed, Current Tier, Days to Showcase (formula)
Row 5 — All clear list (greens on track)
A count or simple list of Green tier implementations with on-time pulse submissions this week. This section exists specifically to confirm the system is working — silence here is a warning sign, not a comfort.
Count of Green implementations with Last Pulse Submission Date in current week · Link to full green list view
Zapier automation specifications — Quick Wins (build for fall)

Each automation below is a complete specification ready for a Zapier builder. Sorted by build priority. Total estimated build time: 20–25 hours for all 10.

1
Trigger: Airtable record created with Implementation Status = Committed
Actions: (1) Send welcome email to Instructor Email with onboarding intro + A360 Part 1 link. (2) Send stakeholder mapping form link to same email. (3) Create a Slack notification to Ivan: "New commitment: [School Name] — record created."
Build first — this is Stage 1's only automation and the starting point for the entire chain.
Low effort
2
Trigger: Reach 360 course.completed webhook fires for a teacher (course = Part 1)
Actions: (1) Find matching Airtable record by teacher email. (2) Update Part 1 Completion Date field with webhook timestamp. (3) Send email to teacher with Calendly link to select Part 2 window.
Reach 360 webhook is confirmed supported. The Zapier "Reach 360: Course Completed" trigger is the native integration — no custom webhook needed.
Low effort
3
Trigger: Calendly invitee created (teacher books Part 2 window)
Actions: (1) Find matching Airtable record by invitee email. (2) Update Part 2 Window Selected date field with the booked event date.
Calendly has a native Zapier integration. Map the invitee email to the Instructor Email field to find the right Airtable record.
Low effort
4
Trigger: Google Form submission (weekly pulse form)
Actions: (1) Create a new record in the Weekly Pulse Log table with: Implementation Key (matched from form "which school" dropdown), Submission Date (timestamp), Session Number, Pulse Score, Open Text Response, Module Deliverable Submitted. (2) Update the parent Program Implementation record: Current Session, Last Pulse Submission Date, Latest Pulse Score. (3) If score ≤ 2 AND session between 6–15: set Open Flag = true on parent record. (4) If Module Deliverable Submitted = false AND session is a module-end session: set Open Flag = true + Module Deliverable Status = Missing.
Most complex automation in the system — multiple actions in one Zap. Use Zapier's multi-step Zap. The "which school" dropdown on the form must use Implementation Keys (or exact school names that match Airtable records) to avoid matching errors.
Med effort
5
Trigger: Zapier Schedule — every Friday at 5pm
Actions: (1) Search Airtable for all active implementations where Last Pulse Submission Date is more than 6 days ago. (2) For each match: send reminder email or text to Instructor (or add to a reminder list sent to Ivan to follow up manually in v1). (3) Set Open Flag = true if second consecutive miss (check if previous Friday also had no submission).
This is the missed-submission detection automation. In v1, the "send text" may need to be handled manually by Ivan using the reminder list. SMS automation depends on texting tool decision (see Deliverable 5).
Med effort
6
Trigger: Airtable record updated — Current Session field changes to 6, 11, 16, or 20
Actions: (1) Identify which module is being entered (6=M2, 11=M3, 16=M4, 20=M5). (2) Send the corresponding JIT Loom kit email to Instructor Email. (3) Log kit send timestamp in Airtable.
Four separate Zaps (one per threshold) or one multi-path Zap using Zapier's Paths feature. The JIT Loom kit emails must be pre-written and linked before this Zap can be activated.
Med effort
7
Trigger: Airtable record updated — Showcase Date field is set AND today's date = Showcase Date minus 21 days
Actions: (1) Send client re-confirmation email to client contact (from MI Client Partner linked record). (2) Update Implementation Status to "Showcase Prep." (3) Send Ivan a digest: "Showcase prep begins today for [School] — showcase in 21 days."
Use Zapier's date filter or a Zapier Schedule that runs daily and checks which records have Showcase Date = today + 21. The daily-check pattern is more reliable than trying to trigger on a specific future date.
Low effort
8
Trigger: Zapier Schedule — daily check: Showcase Date = today
Actions: (1) Send student survey link to Instructor Email with instructions for in-class facilitation. (2) Send teacher feedback form link to Instructor Email. (3) Send client debrief scheduling link + pre-conversation form to client contact email. (4) Create calendar invite for Ivan: "Client debrief due by [Showcase Date + 5 days] — [School]."
Run daily; only fires when Showcase Date = today. Multiple emails send on showcase day — consider staggering by a few hours so they don't all arrive in the teacher's inbox simultaneously.
Low effort
9
Trigger: Zapier Schedule — Monday at 8am
Actions: (1) Pull all active implementations from Airtable (Status = Active or Showcase Prep). (2) Sort by Current Tier (Red first). (3) Format as a digest email showing: School | Session | Tier | Last Pulse Date | Open Flag | Next milestone. (4) Send to Ivan's email.
The digest email formatting is the hardest part of this automation — Zapier's email body supports basic HTML. Use a formatter step to build the table. In H2, the admin Chipp Bot delivers this digest in plain language with site-specific context.
Med effort
10
Trigger: Airtable record updated — Renewal Disposition = Yes OR Maybe
Actions: (1) Schedule a 30-day follow-up email reminder to Ivan: "Follow up on [School] renewal — disposition: [Yes/Maybe]." (2) Schedule a 60-day follow-up if the 30-day one goes unresolved (check if Renewal Disposition still = Maybe at day 30).
Use Zapier's Delay step to schedule the 30-day reminder. The 60-day conditional check requires a second Zap or a Zapier workflow with a decision branch.
Low effort
Build sequence recommendation

Build in this order: (1) Add new fields to Program Implementation table. (2) Create Weekly Pulse Log table with linking. (3) Build Zaps 1–3 (commitment and training sequence). (4) Build Zap 4 (pulse form routing — the most complex). (5) Build Zaps 5–6 (missed submission detection + JIT kits). (6) Build Zaps 7–8 (showcase triggers). (7) Build Zap 9 (Monday digest). (8) Build Zap 10 (renewal follow-ups). (9) Build Airtable Interface (Control Center). Test with 2 synthetic records before piloting with real schools.

Build sequencing assumes the pulse form, pacing mapper, JIT Loom kits, and Part 2 Calendly windows are all created before Zaps 3–6 can be activated. Content must precede automation.
